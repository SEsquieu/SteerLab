# Manifesto

AI-assisted coding is normal now.

Engineering evaluation should change accordingly.

For a long time, interview systems have leaned on convenient proxies: code puzzles, isolated implementation tasks, and artificial constraints that reward speed more than judgment. Those methods never captured the whole job. In the AI era, they capture even less.

Raw code production is no longer the only signal, and in many contexts it is not the most important one. Strong tools can generate code quickly. They cannot remove the need for judgment about what should be built, what is broken, what is risky, what is missing, and what should not be trusted.

The important question is shifting.

Not simply:

Can this person produce code unaided, on demand, in a vacuum?

More often:

Can this person understand systems, reason under constraints, direct powerful tools responsibly, and move a problem toward a sound outcome?

SteerLab frames that shift through three challenge archetypes.

## Broken System Investigation

Much of engineering is not greenfield creation. It is entering a system that already exists, understanding how it fails, forming hypotheses from partial evidence, and making disciplined decisions under uncertainty.

This is where debugging, systems reasoning, failure mode awareness, and communication clarity become visible. It is difficult to fake. It is also difficult to measure with a puzzle.

## Architecture Thought Experiment

Another large share of engineering work is design under constraints. Requirements are partial. Systems have history. Dependencies have costs. Tradeoffs are real. The task is not to produce the one true answer. The task is to define a sound approach, expose its assumptions, and reason clearly about boundaries, risks, and consequences.

This is where decomposition, architecture reasoning, maintainability thinking, and tradeoff judgment show up.

## Tool Steering Challenge

AI tools change the shape of implementation work, but they do not remove the need for engineering competence. They relocate some of it.

When an engineer works with a powerful model, value increasingly depends on:

- providing the right context
- breaking the task into safe increments
- asking good questions
- setting explicit acceptance criteria
- validating outputs against system behavior and constraints
- rejecting plausible but wrong answers

This is not lesser engineering. It is a new surface area for engineering judgment.

These three archetypes are not the whole field, but together they describe a large portion of modern software work:

- understanding and debugging existing systems
- designing under constraints
- using leverage responsibly

That is why they sit at the center of this project.

SteerLab also treats those archetypes as shared foundations rather than single-use interview content. The same core scenarios can support both evaluation and training:

- evaluation asks how well someone reasoned
- training asks how someone can learn to reason better

That split matters because the project is not only about measuring judgment. It is also about helping engineers build better habits around debugging, architecture, validation, and AI-assisted work.

## What This Project Believes

SteerLab takes several positions seriously.

Engineering value is increasingly found in judgment.

That includes diagnosis, boundary definition, tradeoff analysis, validation discipline, and the ability to steer tools rather than merely operate them.

AI use should be evaluated openly, not ignored.

Banning AI from evaluation often makes the exercise less realistic. Pretending AI changes nothing makes the exercise less honest. We need formats that examine how engineers actually use leverage, including how they verify and constrain it.

AI use should also be taught responsibly, not treated as a shortcut.

Teams are increasingly exposed to a failure mode where generated output moves faster than engineering judgment. Better tooling alone will not fix that. Engineers need practice in constraining model scope, validating generated work, and reasoning about system consequences before shipping.

Open-source exploration is the right medium.

Evaluation frameworks shape hiring behavior. Hiring behavior shapes careers. That makes opacity a poor default. Challenge formats, rubrics, and assumptions should be visible, debated, and improved in public.

There is no finished standard yet.

This repository is a draft with a point of view. It is not a decree. It does not claim one universal rubric, one universal score, or one perfect hiring loop. It is an attempt to create shared artifacts serious enough to argue over.

## What This Project Rejects

- reducing engineering competence to raw code output
- treating AI abstinence as the default measure of skill
- using unrealistic puzzles as a stand-in for systems work
- building a closed, opaque definition of the future standard
- mistaking fluent AI usage for sound engineering judgment

SteerLab exists to make one question concrete:

How well can someone steer powerful tools, systems, and decisions toward good engineering outcomes?

If that is becoming central to the job, then it deserves better evaluation formats than the ones we inherited.
